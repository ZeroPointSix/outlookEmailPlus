# Outlook 邮件管理工具｜批量操作与自动化增强 PRD

- 文档状态：草案
- 版本：V1.0
- 日期：2026-02-24
- 负责人：待定

---

## 1. 背景与现状

本项目已具备多邮箱管理、邮件查看、分组/标签、批量操作等核心能力。随着用户使用场景的深入，暴露出以下产品层面的需求：

- **批量操作效率**：用户需要频繁修改大量邮箱信息（移动分组、打标签等），但缺少快速全选功能，操作效率低。
- **注册自动化**：用户在使用邮箱进行服务注册时，需要手动查看邮件、复制验证码或链接，流程繁琐且容易出错。
- **邮件监控**：用户需要实时监控邮箱是否收到新邮件（特别是验证码邮件），但目前需要手动刷新，无法自动通知。

本 PRD 聚焦"批量操作与自动化增强"的产品需求，目标是提升用户在批量管理和自动化注册场景下的操作效率。

---

## 2. 目标（What）

### 2.1 业务目标

- **提升批量操作效率**：通过全选功能，让用户能快速选中当前分组下的所有邮箱，配合现有批量操作（移动分组、打标签等）提升效率。
- **简化注册流程**：通过智能提取验证码和链接，让用户一键复制所需信息，减少手动查找和复制的时间。
- **实现邮件自动监控**：通过定时轮询机制，自动检测新邮件并通知用户，无需手动刷新。

### 2.2 成功指标（可验收）

- 用户可以通过全选功能快速选中当前分组下的所有邮箱，并执行批量操作。
- 用户可以通过复制按钮一键获取最新邮件中的验证码或链接，成功率 > 90%。
- 用户可以配置自动轮询，系统能按设定间隔和次数自动检测新邮件并通知。

---

## 3. 范围与非目标

### 3.1 本次范围

- 新增全选功能，支持快速选中当前分组下的所有邮箱。
- 新增邮件信息快速复制功能，智能提取验证码和链接。
- 新增自动轮询功能，支持定时检测新邮件并通知用户。

### 3.2 非目标（本次不做）

- 不支持跨分组全选（只支持当前分组）。
- 不支持自定义验证码提取规则（使用预设的智能识别规则）。
- 不支持多邮箱同时轮询（只轮询当前选中的邮箱）。

---

## 4. 用户与场景

### 4.1 用户画像

- **管理员/操作者（单人或小团队）**
  - 需要批量管理多个邮箱账号
  - 需要使用邮箱进行大量服务注册
  - 需要快速获取邮件中的验证码或激活链接
  - 需要实时监控邮箱是否收到新邮件

### 4.2 关键使用场景（User Stories）

- 作为管理员，我希望能快速全选当前分组下的所有邮箱，然后批量移动到其他分组或打标签。
- 作为管理员，我希望在注册服务时，能一键复制最新邮件中的验证码，无需手动查找和复制。
- 作为管理员，我希望在等待验证邮件时，系统能自动检测并通知我，无需手动刷新。
- 作为管理员，我希望能配置轮询间隔和次数，避免过度请求导致账号异常。

## 5. 功能需求

> 说明：以下为"需求定义"，不讨论具体实现方式。

### 5.1 全选功能

#### 5.1.1 功能描述
- 在邮箱面板的工具栏中，排序按钮的右边，增加一个全选复选框。
- 点击全选复选框时，自动选中当前分组下的所有邮箱。
- 再次点击全选复选框时，取消所有选中。

#### 5.1.2 交互细节
- 全选复选框的状态与邮箱列表的选中状态联动：
  - 当所有邮箱都被选中时，全选复选框显示为选中状态
  - 当部分邮箱被选中时，全选复选框显示为半选状态（indeterminate）
  - 当没有邮箱被选中时，全选复选框显示为未选中状态
- 全选功能支持与现有批量操作配合使用：
  - 批量移动分组
  - 批量打标签/去标签

#### 5.1.3 边界条件
- 全选只影响当前分组下的邮箱，不跨分组。
- 切换分组时，全选状态重置。
- 搜索或筛选后，全选只影响当前显示的邮箱。

---

### 5.2 邮件信息快速复制功能

#### 5.2.1 功能描述
- 在邮箱列表中，每个邮箱账号名称的右边，增加一个复制按钮（图标：📋）。
- 点击复制按钮时，自动获取该邮箱的最新邮件，并智能提取验证码和链接。
- 提取成功后，自动复制到剪贴板，并显示成功提示。

#### 5.2.2 提取规则

**验证码提取规则（优先级从高到低）：**
1. **智能识别**：识别邮件中"验证码"、"code"、"验证"、"verification"等关键词附近的数字/字母组合（4-8位）
2. **保底提取**：如果智能识别失败，提取邮件中所有连续的4-8位数字+字母组合，取第一个

**链接提取规则：**
- 提取邮件中所有 HTTP/HTTPS 链接

**输出格式：**
- 如果同时存在验证码和链接：`111111 https://link1.com https://link2.com`（空格分隔）
- 如果只有验证码：`111111`
- 如果只有链接：`https://link1.com https://link2.com`
- 如果都没有：显示"未找到验证信息"提示

#### 5.2.3 文件夹策略
- 同时从收件箱和垃圾邮件文件夹获取邮件
- 按时间排序，取最新的一封邮件
- 不包括已删除邮件文件夹

#### 5.2.4 交互反馈
- **成功**：右上角弹出"已复制"通知（2秒后自动消失）
- **失败**：显示"未找到验证信息"提示
- **无数据**：自动触发一次获取邮件操作，获取完成后自动复制
- **获取中**：复制按钮显示加载状态，禁止重复点击

#### 5.2.5 边界条件
- 如果邮箱从未获取过邮件，点击复制按钮时自动触发获取邮件操作。
- 如果邮箱获取邮件失败，显示具体失败原因（如"Token 过期"、"网络错误"等）。
- 如果邮件内容为空或无法解析，显示"未找到验证信息"。

---

### 5.3 自动轮询与新邮件通知

#### 5.3.1 功能描述
- 在设置中心增加"自动轮询"配置项，支持启用/禁用。
- 启用后，当用户选中一个邮箱时，自动开始轮询该邮箱的新邮件。
- 轮询到达设置次数后自动停止。
- 检测到新邮件时，通过页面通知和视觉提示告知用户。

#### 5.3.2 配置项
- **启用自动轮询**：开关，默认关闭
- **轮询间隔**：单位秒，默认 10 秒，用户可自定义（建议范围：5-300 秒）
- **轮询次数**：默认 5 次，用户可自定义（建议范围：1-100 次，或支持无限轮询）

#### 5.3.3 轮询行为
- **轮询范围**：只轮询当前选中的邮箱
- **轮询触发**：当用户在邮箱列表中选中一个邮箱时，如果启用了自动轮询，则自动开始轮询
- **轮询停止**：
  - 到达设置的轮询次数后自动停止
  - 用户切换到其他邮箱时停止当前轮询
  - 用户关闭页面或退出登录时停止所有轮询
- **轮询逻辑**：
  - 每次轮询时，同时从收件箱和垃圾邮件文件夹获取最新邮件
  - 与上次获取的邮件列表对比，检测是否有新邮件（通过邮件 ID 判断）
  - 如果有新邮件，触发通知

#### 5.3.4 通知方式

**页面内通知：**
- 在页面右上角弹出通知框
- 显示内容：邮箱地址 + 邮件主题
- 示例：`user@outlook.com 收到新邮件：您的验证码是 123456`
- 通知框 5 秒后自动消失，或用户点击关闭

**邮箱列表视觉提示：**
- 在邮箱名称旁边显示红色圆点或数字角标
- 表示该邮箱有未读的新邮件
- 用户点击该邮箱并查看邮件后，红点消失

#### 5.3.5 显示策略
- 轮询过程静默进行，不显示轮询进度或状态
- 只在检测到新邮件时才显示通知
- 不干扰用户的正常操作

#### 5.3.6 边界条件
- 如果轮询过程中获取邮件失败（如 Token 过期、网络错误），停止轮询并显示错误提示。
- 如果用户在轮询过程中手动点击"获取邮件"按钮，重置轮询计数器。
- 如果用户在设置中修改了轮询配置，下次选中邮箱时生效。

## 6. 非功能需求

### 6.1 性能
- 全选操作应在 100ms 内完成，即使邮箱数量达到 1000 个。
- 复制按钮点击后，应在 2 秒内完成验证码/链接提取（不包括获取邮件的时间）。
- 轮询操作不应影响页面的正常使用，不应导致页面卡顿或无响应。

### 6.2 可靠性
- 验证码提取的成功率应 > 90%（基于常见邮件格式）。
- 轮询过程中如果出现网络错误或 API 失败，应自动停止轮询并提示用户。
- 复制操作失败时，应给出明确的失败原因和建议操作。

### 6.3 可用性
- 全选复选框的状态应与邮箱列表的选中状态实时同步。
- 复制按钮应有明确的视觉反馈（加载状态、成功状态、失败状态）。
- 轮询通知应清晰易懂，不应过度打扰用户。

### 6.4 安全
- 复制到剪贴板的内容不应包含敏感信息（如完整的 Token、密码等）。
- 轮询请求应遵循现有的 API 优先级回退策略（Graph API → IMAP 新 → IMAP 旧）。
- 轮询频率应有合理限制，避免过度请求导致账号被封禁。

## 8. 附录

### 8.1 验证码提取示例

**示例 1：智能识别**
```
邮件内容：您的验证码是 123456，请在 10 分钟内完成验证。
提取结果：123456
```

**示例 2：保底提取**
```
邮件内容：Please use code ABC123 to verify your account.
提取结果：ABC123
```

**示例 3：同时存在验证码和链接**
```
邮件内容：您的验证码是 123456，或点击链接激活：https://example.com/verify?token=xxx
提取结果：123456 https://example.com/verify?token=xxx
```

**示例 4：只有链接**
```
邮件内容：请点击以下链接完成验证：https://example.com/verify
提取结果：https://example.com/verify
```

### 8.2 轮询配置示例

**场景 1：快速注册**
- 轮询间隔：10 秒
- 轮询次数：5 次
- 说明：适合快速注册场景，总共轮询 50 秒

**场景 2：长时间等待**
- 轮询间隔：30 秒
- 轮询次数：20 次
- 说明：适合需要长时间等待的场景，总共轮询 10 分钟

**场景 3：持续监控**
- 轮询间隔：60 秒
- 轮询次数：无限（或设置为 999）
- 说明：适合需要持续监控的场景，直到用户手动停止

## 9. 风险与依赖

### 9.1 风险
- **验证码识别准确率**：不同邮件格式差异较大，可能导致识别失败率较高。
  - 缓解措施：提供保底提取规则，并支持用户手动查看邮件内容。
- **轮询频率限制**：过高的轮询频率可能导致 API 限流或账号被封。
  - 缓解措施：设置合理的默认值（10 秒间隔），并在设置中提示用户。
- **浏览器剪贴板权限**：部分浏览器可能限制剪贴板访问。
  - 缓解措施：使用标准的 Clipboard API，并在失败时提示用户手动复制。

### 9.2 依赖
- 依赖现有的邮件获取 API（Graph API / IMAP）
- 依赖现有的通知组件（页面内通知）
- 依赖现有的批量操作功能（移动分组、打标签）
