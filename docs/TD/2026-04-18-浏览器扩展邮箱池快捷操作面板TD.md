## 1. 技术目标

1. **可安装性**：扩展可本地加载（开发者模式），后续可打包为 `.crx`
2. **动态服务器支持**：用户填入任意自托管地址后，扩展能正常发起跨域请求，不被 Chrome 拦截
3. **状态持久化**：跨 Popup 开关保留当前任务与历史记录
4. **Popup 生命周期安全**：用户关闭 Popup 不导致数据丢失，重新打开可恢复任务
5. **主应用最小改动**：仅补充 CORS 来源，不新增接口、不改数据库

## 2. 技术难点与解决方案

### 2.1 【最大难点】动态服务器地址 vs Manifest V3 host_permissions

#### 问题描述

Chrome Manifest V3 要求扩展在 `manifest.json` 的 `host_permissions` 字段中**静态声明**可访问的主机。若扩展向某个主机发起 fetch 请求，但该主机未在 `host_permissions` 中声明，Chrome 会在请求发出前就拦截并抛出：

```
Access to fetch at 'https://user-server.com/api/...' from origin 'chrome-extension://xxx'
has been blocked by CORS policy: No 'Access-Control-Allow-Origin' header is present
```

与服务端 CORS 不同，这是 Chrome 在**客户端侧**的权限检查，服务端无从干预。

本扩展核心场景：用户在设置面板自由填写服务器地址，在安装时无法预知目标域名。

#### 方案 A：`<all_urls>`（简单但权限过宽）

```json
{
  "host_permissions": ["<all_urls>"]
}
```

- **优点**：开箱即用，无需动态申请
- **缺点**：Chrome 会在安装时警告「可以读取和更改所有网站上的数据」，且发布到 Chrome 应用商店时需要 Google 额外审查

#### 方案 B：`optional_host_permissions` + 动态申请（推荐）

```json
{
  "optional_host_permissions": ["<all_urls>"]
}
```

在设置面板**保存**时，调用：

```javascript
async function requestPermissionForHost(serverUrl) {
  const url = new URL(serverUrl);
  const origin = `${url.protocol}//${url.hostname}/*`;
  const granted = await chrome.permissions.request({ origins: [origin] });
  return granted;
}
```

- **优点**：仅在用户主动填写地址时申请权限，Chrome 弹出授权对话框，语义清晰
- **缺点**：用户首次保存配置时需点击确认；`chrome.permissions.request` 必须在用户操作回调中调用（不能在 load 阶段异步调用）

#### 选定方案

**本 TD 采用方案 B（optional_host_permissions）**，理由：
1. 避免过宽权限警告，未来上架应用商店阻力更小
2. 用户授权意图明确（在「保存配置」操作时触发）
3. 开发阶段可临时使用方案 A 进行调试，生产版切换方案 B

#### 主应用 CORS 仍需配置

即使 Chrome 客户端权限通过，服务端 CORS 仍需响应正确的 `Access-Control-Allow-Origin`。两者缺一不可：

| 层级 | 负责方 | 配置内容 |
|------|--------|---------|
| 客户端权限 | Chrome 扩展 manifest | `optional_host_permissions: <all_urls>` |
| 服务端 CORS | Flask 主应用 | 允许 `chrome-extension://` 来源 |

### 2.2 Popup 生命周期 vs 长等待请求（wait=60）

#### 问题描述

Manifest V3 Popup 没有持久化的 Background Service Worker（本扩展主动选择不用 SW）。当用户关闭 Popup 窗口时，所有 Popup 内的 JavaScript 执行上下文立即销毁，包括正在进行中的 `fetch` 请求。

获取验证码接口（`wait=60`）的服务端等待时间最长 60 秒。若用户在等待期间关闭了 Popup，fetch 被中断，验证码拿不到，用户需要重新打开 Popup 并再次点击「获取验证码」。

#### 解决方案

**本 TD 接受这一限制，采用以下策略缓解**：

1. **UI 提示**：进入 `fetching` 状态时，面板显示「⚠️ 等待验证码期间请勿关闭面板」
2. **超时提示**：若 fetch 被中断（`AbortError`），显示「已中断，可重试」而非「失败」
3. **任务保留**：currentTask 在 `fetching` 状态下写入 storage，Popup 重新打开时恢复到 `claimed` 状态（验证码字段为 null），用户可直接再次点击「获取验证码」
4. **AbortController 超时兜底**：

```javascript
const controller = new AbortController();
const timeoutId = setTimeout(() => controller.abort(), 65000); // 65s 兜底
try {
  const resp = await fetch(url, { signal: controller.signal });
  // ...
} catch (err) {
  if (err.name === 'AbortError') {
    showMessage('等待超时，请重试');
  } else {
    showMessage('网络错误：' + err.message);
  }
} finally {
  clearTimeout(timeoutId);
}
```

#### 不考虑 Background SW 的原因

引入 Service Worker 会带来：
- 额外的 `"background"` 配置和 `service_worker.js` 文件
- SW 自身有休眠机制（Chrome 会在 30s 不活跃后终止 SW），维持持久连接需要心跳
- 消息通信（Popup ↔ SW）增加代码复杂度

收益（规避用户关闭 Popup 中断请求）与成本不成比例，本 TD 明确不引入。

### 2.3 Manifest V3 内容安全策略（CSP）

#### 问题描述

Manifest V3 的 CSP 默认禁止：
- 内联 JavaScript（`<script>` 标签内直接写代码）
- `eval()` 和 `new Function()`
- 内联事件处理器（`onclick="..."` 等 HTML 属性）

当前 `browser-extension/popup.html` 预览版大量使用内联 `<script>` 标签和 `onclick` 属性，**无法直接作为正式扩展文件使用**。

#### 解决方案

正式版 `popup.html` 须满足：

1. **所有 JS 移入外部文件**：
   ```html
   <!-- popup.html 中只引用外部文件 -->
   <script src="popup.js"></script>
   ```

2. **事件绑定改为 JS 内 addEventListener**：
   ```javascript
   // popup.js
   document.getElementById('btn-claim').addEventListener('click', handleClaim);
   ```

3. **保留内联 CSS 或引用外部 CSS**（CSS 无限制）

#### 预览版与正式版文件定位

| 文件 | 定位 | 是否可直接用于扩展 |
|------|------|------------------|
| `browser-extension/popup.html`（当前） | UI 交互原型，纯浏览器预览 | ❌（含内联 JS + 预览工具栏） |
| `browser-extension/popup.html`（正式版） | 扩展 Popup 页面，仅 DOM 结构 + 外部 JS 引用 | ✅ |
| `browser-extension/popup.js`（待创建） | 所有交互逻辑 | ✅ |

实现阶段：先保留预览版（改名为 `popup.preview.html`），再创建生产版 `popup.html`。

### 2.4 Extension ID 不稳定问题

#### 问题描述

Chrome 本地加载（开发者模式）的扩展，每次「移除再重新加载」时，Extension ID 会重新生成。这导致 `chrome-extension://<id>` 中的 `<id>` 部分每次不同。

若主应用 CORS 白名单写死具体 ID，换个环境或重装后就会失效。

#### 解决方案

服务端 CORS 配置**匹配 `chrome-extension://` 协议前缀**，而非写死具体 ID：

```python
# outlook_web/app.py 或 CORS 中间件

ALLOWED_ORIGINS_REGEX = [
    r"^chrome-extension://.*$",
    r"^http://localhost(:\d+)?$",
    r"^https://.*$",
]
```

或使用 `flask-cors` 的 `origins` 列表接受正则表达式：

```python
from flask_cors import CORS

CORS(app,
     resources={r"/api/external/*": {"origins": [
         re.compile(r"chrome-extension://.*"),
         # 其他已有来源
     ]}},
     supports_credentials=False,
     allow_headers=["X-API-Key", "Content-Type"]
)
```

> 安全说明：`chrome-extension://` 前缀只在 Chrome 扩展环境中出现，普通网页无法伪造该 Origin，允许此前缀不会扩大实际攻击面。

### 2.5 主应用 CORS 改动技术实现

#### 当前 CORS 现状（需要调研）

需要确认主应用当前 CORS 配置位置。基于架构文档，关键位置为 `outlook_web/app.py` 的 `create_app()` 工厂函数。

#### 改动方案

在 `create_app()` 中，对 `/api/external/*` 路径增加 `chrome-extension://` 来源支持：

```python
# 在已有的 CORS 配置基础上，补充来源
# 注意：flask-cors 的 origins 支持列表，也支持正则
from flask_cors import CORS
import re

CORS(app, resources={
    r"/api/external/*": {
        "origins": [
            re.compile(r"^chrome-extension://.*$"),
            # 保留已有来源规则
        ],
        "methods": ["GET", "POST", "OPTIONS"],
        "allow_headers": ["Content-Type", "X-API-Key"],
    }
})
```

如果当前主应用对 `/api/external/*` 没有 CORS 配置（全局不配置），则需要先确认实际请求来源是否已存在限制，再决定补充范围。

#### 改动影响评估

- 仅影响 `/api/external/*` 路由
- 不影响 Session 鉴权的内部 API（`/api/accounts/*` 等）
- `chrome-extension://` 来源无法伪造，不扩大现有安全边界
- `OPTIONS` 预检请求需正确返回 `Access-Control-Allow-*` 响应头

### 2.6 chrome.storage.local 操作原子性

#### 问题描述

`chrome.storage.local.set()` 和 `chrome.storage.local.get()` 都是异步的。在连续多次写入时（如：申领成功后写入 `currentTask`，同时追加一条「开始」历史记录），若不正确 await，可能导致数据覆盖。

#### 解决方案

封装统一的 storage 读写助手：

```javascript
// storage.js（被 popup.js 引用）
export const Storage = {
  async getAll() {
    return chrome.storage.local.get(['config', 'currentTask', 'history']);
  },
  async setCurrentTask(task) {
    await chrome.storage.local.set({ currentTask: task });
  },
  async clearCurrentTask() {
    await chrome.storage.local.set({ currentTask: null });
  },
  async appendHistory(entry) {
    const { history = [] } = await chrome.storage.local.get('history');
    const next = [entry, ...history].slice(0, 100); // 最多 100 条
    await chrome.storage.local.set({ history: next });
  },
  async getConfig() {
    const { config = {} } = await chrome.storage.local.get('config');
    return config;
  },
  async setConfig(config) {
    await chrome.storage.local.set({ config });
  },
};
```

关键原则：**每次操作先 get 再 set**，不依赖内存缓存直接 set（避免并发写覆盖）。

### 2.7 任务 ID（task_id）的可靠生成

#### 问题描述

- `crypto.randomUUID()` 在 MV3 扩展 Popup 中可直接使用（Chrome 原生 API，无需 polyfill）
- 每次点击「申领」时生成新的 UUID，**生成后立即写入 storage**，不仅依赖内存变量
- 若 Popup 被关闭但 task_id 还未写入 storage，下次打开 Popup 时可能丢失 task_id

#### 解决方案

```javascript
async function claimEmail() {
  const taskId = crypto.randomUUID();
  const callerId = 'browser-extension';
  
  // 先写入 storage，再发 API 请求
  await Storage.setCurrentTask({
    taskId,
    callerId,
    email: null,      // 占位，申领成功后更新
    claimedAt: new Date().toISOString(),
    code: null,
    link: null,
  });

  // 发起 API 请求
  const { serverUrl, apiKey, defaultProjectKey } = await Storage.getConfig();
  const resp = await fetch(`${serverUrl}/api/external/pool/claim-random`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json', 'X-API-Key': apiKey },
    body: JSON.stringify({ caller_id: callerId, task_id: taskId, project_key: defaultProjectKey || undefined }),
  });

  if (!resp.ok) {
    await Storage.clearCurrentTask();  // 申领失败，清空
    throw new Error(await extractErrorMessage(resp));
  }

  const data = await resp.json();
  await Storage.setCurrentTask({ taskId, callerId, email: data.email, claimedAt: new Date().toISOString(), code: null, link: null });
}
```

### 2.8 错误类型区分与用户友好提示

浏览器扩展中 fetch 可能抛出的错误类型：

| 错误类型 | 原因 | 用户提示 |
|---------|------|---------|
| `TypeError: Failed to fetch` | 网络不通 / CORS 拦截 / 服务端不可达 | 「无法连接服务器，请检查地址和网络」 |
| `AbortError` | AbortController 超时或用户关闭 Popup | 「等待超时，可重试」 |
| HTTP 4xx | 业务错误（无邮箱可用 / API Key 错误等） | 读取 `resp.json().message`，原文展示 |
| HTTP 5xx | 服务端异常 | 「服务器内部错误，请稍后重试」 |

注意：CORS 被拦截在扩展侧表现为 `TypeError: Failed to fetch`，与网络不通相同，无法在前端区分。用户遇到此错误时需引导检查「配置是否正确」以及「服务器是否允许扩展访问」。

## 5. 主应用 CORS 改动详细方案

### 5.1 调研现状

在实现阶段，需要先确认：

1. `outlook_web/app.py` 中是否已有 `flask-cors` 或手动 CORS 中间件
2. 当前 `/api/external/*` 路由的 CORS 配置状态
3. 是否有针对特定来源的白名单逻辑

### 5.2 改动实施

根据调研结果，选择以下之一：

**场景 A：`flask-cors` 全局配置**
```python
# 在 CORS(app, ...) 的 origins 列表中追加
import re
origins = [...existing..., re.compile(r"^chrome-extension://.*$")]
```

**场景 B：无全局 CORS，手动在 Blueprint 中处理**
```python
# external_pool.py Blueprint 中
@bp.after_request
def add_cors_headers(response):
    origin = request.headers.get('Origin', '')
    if origin.startswith('chrome-extension://') or is_allowed_origin(origin):
        response.headers['Access-Control-Allow-Origin'] = origin
        response.headers['Access-Control-Allow-Headers'] = 'Content-Type, X-API-Key'
        response.headers['Access-Control-Allow-Methods'] = 'GET, POST, OPTIONS'
    return response
```

**场景 C：全局没有 CORS 限制（当前调用方都在同域）**
- 补充 `flask-cors` 仅对 `/api/external/*` 生效
- 只允许 `chrome-extension://` 和原有合法来源

## 6. 验收口径（TD 视角）

1. 本地加载扩展后，设置面板保存服务器地址时触发权限申请弹窗，用户授权后可正常发起请求
2. 申领时 `task_id` 立即写入 storage，Popup 异常关闭重开后 currentTask 仍存在
3. 获取验证码期间关闭 Popup 再打开，状态恢复为 `claimed`（可重试）
4. 完成/释放时即使 API 失败，currentTask 仍被清空，历史记录标注 `apiError: true`
5. 主应用 CORS 更新后，浏览器 DevTools 中扩展发出的请求不再报 CORS 错误
6. 正式版 `popup.html` 不含任何内联 JavaScript，CSP 检查通过
7. manifest.json 中 `permissions` 只含 `storage`、`tabs`、`windows`，不含 `<all_urls>` 静态声明；`windows` 权限用于 `chrome.windows.create` 弹出独立可拖拽浮窗
8. 历史记录超过 100 条时，最旧的被自动丢弃，storage 不超出配额
