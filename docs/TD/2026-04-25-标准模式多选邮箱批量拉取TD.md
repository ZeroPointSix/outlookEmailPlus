## 1. 文档目标

本文档解决以下技术实现问题：

1. 标准模式“批量拉取邮件”按钮应该落在哪个前端文件与交互链路中
2. 如何在不新增后端批量接口的前提下，复用现有单账号邮件接口完成 latest-only 批量拉取
3. 如何在当前状态模型下回写缓存，而不破坏 `currentAccount` 单账号心智
4. 在单 worker / 单进程约束下，如何保证 V1 的可用性
5. 后续若扩展为异步任务，应如何从当前实现平滑演进

**本文档不解决的问题**：

- 多账号混合邮件列表
- 自动切换当前账号
- 紧凑模式同步入口
- 单次选择过大的性能优化细节（如任务切片、后台 job 化）

## 2. 当前技术现状

### 2.1 批量操作状态归属

当前标准模式批量操作主状态位于 `static/js/main.js`：

1. `selectedAccountIds` — 已选账号集合
2. `getActiveAccountCheckboxes()` — 当前可操作的复选框集合
3. `updateBatchActionBar()` — 控制批量操作栏显示
4. `showBatchRefreshConfirm()` / `showBatchDeleteConfirm()` — 现有批量动作入口

结论：

- **新的“批量拉取邮件”入口应与现有批量操作保持同一归属，落在 `main.js`**

### 2.2 账号与邮件缓存归属

当前相关状态包括：

| 状态 | 所在位置 | 作用 |
|------|----------|------|
| `accountsCache` | `static/js/main.js` | 按分组缓存账号列表 |
| `currentAccount` | `static/js/main.js` | 当前右侧查看的账号 |
| `currentFolder` | `static/js/main.js` | 当前右侧查看的 folder |
| `currentEmails` | `static/js/main.js` | 当前右侧邮件列表 |
| `emailListCache` | `static/js/main.js` | 按 `email + folder` 缓存邮件列表 |
| `syncAccountSummaryToAccountCache` | `static/js/features/groups.js` | 将 `account_summary` 回写账号缓存 |

结论：

- V1 无需新增独立缓存结构，应继续复用 `emailListCache`
- 若接口返回 `account_summary`，应继续调用 `syncAccountSummaryToAccountCache(...)`

### 2.3 现有接口形态

当前邮件读取接口为单账号模式：

`GET /api/emails/<email_addr>`

特征：

1. 支持 `folder` / `skip` / `top`
2. 返回 `emails` / `method` / `has_more`
3. 成功时可能返回 `account_summary`

结论：

- V1 可直接复用该接口完成批量编排，不必先补后端 selected fetch API

### 2.4 单 worker 约束

项目当前存在单 worker / 单进程约束，相关设计已在仓库其他模块中明确使用。

对本功能的影响：

1. 不适合一开始就做后端长时间阻塞式批量接口
2. 不适合默认高并发风暴式拉取
3. V1 更适合走前端轻量编排 + latest-only 请求

## 3. 核心技术决策

### 3.1 决策一：V1 不新增后端批量接口

**选定方案**：前端直接编排现有 `GET /api/emails/<email_addr>`

**理由**：

1. 现有批量动作都在 `main.js` 内完成编排，风格一致
2. 当前状态模型已经足够承接缓存写回
3. 可避免在单 worker 下引入新的阻塞式批量链路

### 3.2 决策二：批量动作落点放在 `main.js`

**选定方案**：按钮入口在 `templates/index.html`，处理函数放在 `static/js/main.js`

**理由**：

1. `selectedAccountIds` 与 `batchActionBar` 控制逻辑都在 `main.js`
2. 与现有 `showBatchRefreshConfirm()` / `batchDeleteAccounts()` 放在同一层，更符合当前前端组织方式
3. 避免把批量行为拆散到 `groups.js` / `emails.js`

### 3.3 决策三：latest-only 固定为 `inbox + junkemail`

**选定方案**：每个账号拉取两个 folder，各取 `top=10`

**理由**：

1. 与当前“验证码 / 最新邮件”场景最贴近
2. 用户已明确确认默认范围为收件箱 + 垃圾箱
3. 比整箱同步更适合单 worker 场景

### 3.4 决策四：当前账号不自动切换

**选定方案**：批量完成后保持当前账号不变

**理由**：

1. PRD 已确认“不自动切换当前账号”
2. 当前右侧邮件区仍然是单账号上下文
3. 批量动作应该是预热，不是打断当前浏览

### 3.5 决策五：V1 执行策略优先串行

**选定方案**：默认串行执行；若后续要放开，只允许极小并发

**理由**：

1. 单 worker 架构下，串行是最稳妥的起点
2. 用户当前已确认“先不设选择上限”，因此更需要用低冲击执行策略来保证可用性
3. 若直接做 2×N 并发，很容易退化成“功能有了，但体验变差”

### 3.6 决策六：V2 优先走 job / probe 化

**选定方案**：未来若升级，不优先做阻塞式 SSE，而是优先做 job / probe 化

**理由**：

1. 仓库已有异步探测（probe）思路可借鉴
2. 更符合单 worker 下“任务创建快、后台推进慢”的模型
3. 便于后续做任务状态、失败重试与审计

## 4. 实现详细设计

### 4.1 涉及文件

V1 预计主要改动以下文件：

| 文件 | 作用 |
|------|------|
| `templates/index.html` | 只在标准模式 `batchActionBar` 新增“批量拉取邮件”按钮，不修改 `compactBatchActionBar` |
| `static/js/main.js` | 新增批量拉取入口、执行队列、进度反馈、缓存回写 |
| `static/js/i18n.js` | 新增“批量拉取邮件”等文案词条 |

> V1 预计不需要修改后端 route/controller/service；如后续需要更强任务态，再进入 V2。

### 4.2 前端入口设计

在 `templates/index.html` 标准模式 `batchActionBar` 中新增按钮：

```html
<button class="btn btn-sm btn-ghost" onclick="showBatchFetchConfirm()">批量拉取邮件</button>
```

建议位置：

- 放在“移动分组”和“刷新 Token”之间
- 不扩展到紧凑模式批量栏
- 视觉层级保持为普通动作按钮，避免标准批量栏出现两个 primary 按钮

### 4.3 `main.js` 新增函数

建议新增以下函数：

1. `showBatchFetchConfirm()`
   - 校验 `selectedAccountIds.size > 0`
   - 弹出确认提示
2. `resolveSelectedAccountsForBatchFetch()`
   - 扫描 `Object.values(accountsCache)` 下各分组数组并解析目标账号列表
3. `batchFetchSelectedEmails(accounts)`
   - 串行执行批量 latest-only 拉取
4. `fetchLatestFoldersForAccount(account)`
   - 对单个账号依次拉取 `inbox` 与 `junkemail`
5. `cacheBatchFetchedFolder(email, folder, payload)`
   - 回写 `emailListCache`
6. `refreshCurrentMailboxIfNeeded(email, folder)`
   - 仅当命中 `currentAccount + currentFolder` 时刷新右侧列表

### 4.4 单账号执行流程

伪代码：

```javascript
async function fetchLatestFoldersForAccount(account) {
  const folders = ['inbox', 'junkemail'];
  const result = { success: true, failures: [] };

  for (const folder of folders) {
    try {
      const resp = await fetch(`/api/emails/${encodeURIComponent(account.email)}?folder=${folder}&skip=0&top=10`);
      const payload = await resp.json();

      if (!payload.success) {
        result.success = false;
        result.failures.push({ folder, error: payload });
        continue;
      }

      cacheBatchFetchedFolder(account.email, folder, payload);

      if (payload.account_summary && typeof syncAccountSummaryToAccountCache === 'function') {
        syncAccountSummaryToAccountCache(account.email, payload.account_summary);
      }

      refreshCurrentMailboxIfNeeded(account.email, folder);
    } catch (error) {
      result.success = false;
      result.failures.push({ folder, error });
    }
  }

  return result;
}
```

### 4.5 缓存回写策略

统一写入：

```javascript
emailListCache[`${email}_${folder}`] = {
  emails: sortedEmails,
  has_more: Boolean(payload.has_more),
  skip: 0,
  method: normalizedMethod,
};
```

说明：

1. `skip` 在 V1 latest-only 模式下固定回写为 `0`
2. `emails` 建议仍按现有规则排序后写入缓存
3. 不新增新的批量专用缓存层

### 4.6 当前视图刷新策略

仅当：

1. `email === currentAccount`
2. `folder === currentFolder`

时，才：

1. 更新 `currentEmails`
2. 更新 `emailCount`
3. 调用 `renderEmailList(currentEmails)`

否则只刷新缓存，不刷新右侧 UI。

### 4.7 进度反馈策略

建议复用当前 `main.js` 中已有的持久 Toast 思路（与 Selected 刷新类似）：

1. 开始时显示 `0 / N`
2. 每个账号完成后推进计数
3. 最终展示：
   - 成功数
   - 失败数
   - 失败邮箱摘要

## 5. 风险与缓解

### 5.1 选择规模大时耗时增长

**风险**：

- 用户选择很多账号时，总耗时会明显变长

**缓解**：

1. 保持 latest-only
2. 默认串行，保证稳定
3. 用进度提示管理用户预期

### 5.2 当前账号视图被意外打断

**风险**：

- 批量回写时误刷新当前 UI，导致用户觉得页面“自己跳了”

**缓解**：

- 严格限定只在 `currentAccount + currentFolder` 双命中时刷新右侧列表

### 5.3 单 worker 下新增后端批量接口造成体验退化

**风险**：

- 如果过早引入后端阻塞式 selected fetch，容易把 worker 长时间占住

**缓解**：

- V1 明确不做
- V2 如需升级，优先走 job / probe 化
