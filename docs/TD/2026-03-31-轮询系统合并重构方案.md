# 轮询系统合并重构方案 — 以简洁模式引擎为主

> 编写时间：2026-03-31  
> 前置文档：`docs/BUG/2026-03-30-简洁模式自动轮询-会话BUG记录.md`

---

## 1. 问题陈述

当前存在两套独立的轮询系统：
- **标准模式轮询**（`main.js`，~150 行）：选择账号时自动启动，单账号轮询
- **简洁模式轮询**（`mailbox_compact.js`，~400 行）：复制邮箱时启动，多账号并行

两套系统功能重叠度高（设置结构、错误处理、API 调用），但设置面板分离、代码各自维护，
导致 Bug 难以统一修复（BUG-4/BUG-6）、设置面板混乱（用户困惑"两套轮询开关"）。

**目标**：以简洁模式引擎为主体，将标准模式的轮询功能整合进来，形成**一套统一的轮询引擎**。

---

## 2. 核心差异分析

| 维度 | 标准模式 | 简洁模式 | 合并后方案 |
|------|---------|---------|-----------|
| 触发方式 | 选中账号时自动启动 | 复制邮箱时启动 | **复制邮箱时启动**（两种模式统一行为） |
| 并发 | 单账号 | 多账号并行（Map） | **多账号并行**（Map，沿用简洁模式） |
| API 调用 | 单 folder | inbox + sentitems 并行 | **inbox + sentitems 并行**（更全面） |
| 停止条件 | 次数上限 / 3次错误 | 次数上限 / 3次错误 / 发现新邮件 | **次数上限 / 3次错误 / 发现新邮件** |
| 新邮件处理 | 通知 + 红点 + 更新列表 | 提取验证码 + 复制 | **提取验证码 + 复制**（优先），失败回退到通知 |
| baseline | `knownEmailIds`（来自当前已加载邮件） | `state.baselineIds`（启动时 fetch 构建） | **启动时 fetch 构建 baseline**（更可靠） |
| UI 指示 | 邮箱地址旁小绿点 | 拉取按钮变绿 + 停止监听文字 | **在对应行/卡片上显示统一绿点** |
| 设置 | 独立面板：开关+间隔(5-300s)+次数 | 独立面板：开关+间隔(3-60s)+次数 | **合并为一套设置**：开关+间隔+次数 |
| 可见性处理 | 无 | 后台暂停/前台恢复 | **保留后台暂停/前台恢复** |

---

## 3. 重构方案

### 3.1 架构设计

```
┌─────────────────────────────────────────────┐
│         统一轮询引擎 (poll-engine.js)         │
│                                             │
│  ┌─ pollMap (Map<email, PollState>)        │
│  ├─ startPoll(email, opts)                 │
│  ├─ stopPoll(email, toast?)                │
│  ├─ stopAllPolls()                         │
│  ├─ pollSingleEmail(email, state)          │ ← 核心 poll 逻辑
│  ├─ applyPollSettings(settings)            │
│  ├─ visibilitychange 监听                   │
│  └─ email-copied 事件监听                    │
│                                             │
│  PollState = {                              │
│    timer, startTime, baselineIds,           │
│    errorCount, pollCount, isPolling,        │
│    intervalSec, maxCount, mode              │ ← mode: 'standard'|'compact'
│  }                                          │
└──────────────────────┬──────────────────────┘
                       │
         ┌─────────────┴─────────────┐
         │                           │
   标准模式适配器                简洁模式适配器
   (updateStandardUI)          (updateCompactUI)
   - 小绿点在 .account-email   - 拉取按钮变绿
   - 新邮件通知(toast/badge)   - 停止监听文字
```

### 3.2 设置合并

**before（2 个面板 + 6 个后端字段）**：
```
标准轮询：enable_auto_polling / polling_interval / polling_count
简洁轮询：enable_compact_auto_poll / compact_poll_interval / compact_poll_max_count
```

**after（1 个面板 + 3 个后端字段）**：
```
统一轮询：enable_auto_polling / polling_interval / polling_count
```

- `enable_auto_polling`：总开关，同时控制标准和简洁模式
- `polling_interval`：轮询间隔（秒），范围 3-300
- `polling_count`：最多轮询次数（0=持续），范围 0-100

简洁模式的 3 个专属字段可废弃（后端保留但不再使用，向后兼容）。

### 3.3 触发时机统一

| 场景 | 行为 |
|------|------|
| 标准模式 - 选中账号 | **不自动启动轮询**（改为复制邮箱时启动） |
| 标准模式 - 复制邮箱 | `email-copied` → `startPoll(email)` |
| 简洁模式 - 复制邮箱 | `email-copied` → `startPoll(email)` |

> **注意**：标准模式原来是"选中就轮询"，合并后改为"复制才轮询"，这是一个**行为变更**，
> 需要与用户确认是否可接受。如果需要保留"选中就轮询"，可在标准模式中额外挂钩 `selectAccount` 事件。

## 4. 实施步骤

### Phase 1：提取统一引擎（不改现有功能）
1. 新建 `static/js/features/poll-engine.js`
2. 将 `mailbox_compact.js` 中的轮询引擎代码迁移至此（保留 compact-specific UI 回调注入点）
3. `mailbox_compact.js` 改为调用统一引擎

### Phase 2：标准模式接入统一引擎
1. 删除 `main.js` 中的 `startPolling`/`stopPolling`/`pollForNewEmails` 等（~150 行）
2. 标准模式改为调用统一引擎（注入 standard UI 更新回调）
3. 统一 `showPollingStatusIndicator` / `hidePollingStatusIndicator`

### Phase 3：设置面板合并
1. `index.html`：删除简洁模式独立面板，保留标准轮询面板（调整范围 3-300s）
2. `main.js`：加载/保存只用 `enable_auto_polling`/`polling_interval`/`polling_count`
3. `settings.py`：compact 字段标记为 deprecated，仍可读写但不再新建

### Phase 4：测试更新
1. TC-A：减少为只测标准字段（compact 字段降级为 deprecated 测试）
2. TC-B：更新前端契约测试（只有一个面板）
3. TC-C：引擎测试迁移到 `poll-engine.test.js`
4. 目标：总测试数可能从 49 → 40~45（合并重复测试）

---

## 5. 风险评估

| 风险 | 影响 | 缓解 |
|------|------|------|
| 标准模式"选中就轮询"被移除 | 用户需改为"复制邮箱后轮询" | 询问用户是否需要保留 |
| 大量文件修改 | 可能引入新 Bug | 分 4 个 Phase 逐步执行，每步测试 |
| 后端字段废弃 | 旧配置可能残留 | 保留读取兼容，新配置用标准字段 |
