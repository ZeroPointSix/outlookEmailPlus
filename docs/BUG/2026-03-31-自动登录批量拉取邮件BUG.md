# BUG 报告：自动登录时批量拉取邮件 + 邮件获取失败

**报告日期**：2026-03-31  
**发现人**：用户手动验收  
**严重程度**：🔴 高  
**影响范围**：所有用户的首次登录体验

---

## 一、问题描述

### BUG-1：自动登录进入后强制为所有邮箱拉取邮件

**现象**：用户登录进入系统后，还没有手动选择任何账号，系统就自动对当前分组内**所有**邮箱账号发起了邮件拉取请求。

**预期行为**：首次登录进入时，只应加载分组和账号列表，**不应自动拉取邮件**。邮件拉取应在以下场景触发：
1. 用户**手动选择某个账号**时，加载该账号的邮件
2. 用户**手动开启轮询**后，按配置的间隔周期拉取

### BUG-2：邮件获取失败

**现象**：由于 BUG-1 导致的大量并发请求，部分或全部邮件获取失败（后端可能因为 Token 验证、API 限流等原因拒绝请求）。

**直接原因**：BUG-1 触发了 N 个账号 × 4 个请求 = 4N 个并发 API 调用，后端或微软 Graph API 无法承受这种突发负载。

---

## 二、根因分析

### 调用链路

```
DOMContentLoaded
  → loadGroups()
    → selectGroup(firstNormalGroup.id)         // 首次进入自动选中第一个分组
      → loadAccountsByGroup(groupId)
        → 对所有账号调用 startPoll(acc.email)   // ⚠️ 批量轮询启动
          → 每个 startPoll 发起 2 个 API 请求（inbox + sentitems）建立 baseline
          → 150ms 后再发 2 个请求做首次轮询
```

### 问题代码位置

**文件**：`static/js/features/groups.js`

**第 215-228 行**（`loadAccountsByGroup` 中的批量轮询启动）：
```javascript
// 标准模式：加载分组后，为所有账号启动轮询（跳过已在轮询中的）
var view = typeof mailboxViewMode !== 'undefined' ? mailboxViewMode : 'standard';
if (view !== 'compact' && typeof pollEnabled !== 'undefined' && pollEnabled && typeof startPoll === 'function') {
    var pollCount2 = 0;
    (data.accounts || []).forEach(function(acc) {
        if (acc && acc.email && !(hasPollMap2 && pollMap.has(acc.email))) {
            startPoll(acc.email, { silent: true }); pollCount2++;
        }
    });
    if (pollCount2 > 0 && typeof showToast === 'function') {
        showToast(translateAppTextLocal('已为 ' + pollCount2 + ' 个账号启动轮询监听'), 'info');
    }
}
```

同样的逻辑还出现在**第 175-188 行**（缓存命中路径）。

**文件**：`static/js/features/poll-engine.js`

**第 247-278 行**（`startPoll` 的 baseline + 首次轮询）：
- 每次 `startPoll` 发起 2 个 fetch 请求建立 baseline
- 150ms 后再执行 `pollSingleEmail`（又发 2 个请求）

### 影响计算

如果一个分组有 50 个账号：
- 账号列表加载：1 个 API 请求
- 批量 startPoll：50 × 2 = 100 个 baseline 请求（并发）
- 150ms 后首次轮询：50 × 2 = 100 个轮询请求
- **总计：201 个 API 请求在 1 秒内发出**

## 三、修复方案

### 方案 A（推荐）：首次加载时不启动轮询

在 `loadAccountsByGroup` 中的批量轮询启动逻辑添加条件判断：
- **首次加载**（非 `forceRefresh`）时，**不自动启动轮询**
- 轮询仅在以下场景启动：
  1. 用户手动选择账号后（`selectAccount`）
  2. 用户手动切换分组后（非首次进入）
  3. 用户在设置中开启轮询后（`applyPollSettings`）

具体实现：添加一个 `isFirstLoad` 标志位，首次 `loadGroups` 调用时设为 `true`，`selectGroup` 首次调用时传递该标志，`loadAccountsByGroup` 根据标志决定是否启动轮询。

### 方案 B：延迟轮询启动

保持批量轮询逻辑，但增加**错开延迟**（stagger），每个账号间隔 1-2 秒启动轮询，避免瞬间并发。

### 方案 C：仅轮询当前选中账号

轮询只作用于**当前选中的账号**，而非分组内所有账号。

---

## 四、建议优先级

| 方案 | 改动量 | 用户体验 | 推荐 |
|------|--------|----------|------|
| A：首次不启动轮询 | 小 | ✅ 最佳（按需拉取） | ⭐ 推荐 |
| B：延迟错开 | 中 | 可接受（仍会自动拉取） | 备选 |
| C：仅选中账号轮询 | 小 | ✅ 好（轮询范围缩小） | 可与 A 结合 |

## 六、复现步骤

1. 确保至少有一个分组包含多个邮箱账号
2. 确保轮询功能已开启（设置中 `enable_auto_polling = true`）
3. 打开浏览器开发者工具 → Network 标签
4. 登录系统
5. 观察：进入首页后，立即看到大量 `/api/emails/*` 请求并发发出
6. 部分请求返回错误（如 401 Unauthorized 或超时）
