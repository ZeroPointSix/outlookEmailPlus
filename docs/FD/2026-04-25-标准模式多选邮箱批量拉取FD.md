## 1. 功能定义

### 1.1 核心功能

在标准模式中，为已存在的账号多选能力补上一条新的批量动作：

> 用户勾选多个邮箱后，点击“批量拉取邮件”，系统对这些账号执行 `inbox + junkemail` 的 latest-only 轻量拉取，并将结果写回现有缓存体系。

### 1.2 本期包含

1. 标准模式批量操作栏新增“批量拉取邮件”入口
2. 复用现有 `GET /api/emails/<email_addr>` 拉取两类 folder：
   - `inbox`
   - `junkemail`
3. 复用现有缓存与当前页面状态：
   - `selectedAccountIds`
   - `accountsCache`
   - `emailListCache`
   - `currentAccount`
   - `currentFolder`
4. 前端进度反馈与失败汇总

### 1.3 本期不包含

1. 新的后端阻塞式批量接口
2. 多账号混合邮件列表
3. 自动切换当前账号
4. 紧凑模式同步入口
5. 任务持久化与后台 job 系统（V2 再议）

## 2. 前端交互设计

### 2.1 入口位置

入口位于标准模式 `batchActionBar`。

本期明确：

- 只在标准模式 `batchActionBar` 增加入口
- 不在紧凑模式 `compactBatchActionBar` 增加对应按钮
- 按钮视觉层级采用普通动作样式（ghost），保留“刷新 Token”为唯一 primary 动作

建议按钮顺序：

1. 打标签
2. 去标签
3. 移动分组
4. 批量拉取邮件
5. 刷新 Token
6. 删除

### 2.2 触发条件

仅当以下条件同时满足时允许执行：

1. 当前视图为标准模式
2. `selectedAccountIds.size > 0`

### 2.3 执行前提示

执行前弹出确认，提示信息至少包含：

1. 已选账号数量
2. 本次默认拉取范围：`收件箱 + 垃圾箱`
3. 提示这是 latest-only 轻量拉取，不是整箱同步

### 2.4 执行中反馈

执行中使用前端常驻 Toast 反馈进度：

1. 开始：`正在批量拉取邮件... 0 / N`
2. 进行中：按账号推进计数
3. 完成：展示成功数 / 失败数
4. 部分失败时：展示失败账号列表摘要

### 2.5 执行后交互

1. 当前账号保持不变
2. 不自动切换到第一个或最后一个成功账号
3. 若当前账号刚好在本批次中，且当前显示 folder 与本次已更新缓存一致，则刷新当前右侧邮件列表

## 3. 状态模型与数据流

### 3.1 复用的现有状态

| 状态 | 用途 |
|------|------|
| `selectedAccountIds` | 已选账号集合（跨分组保持） |
| `accountsCache` | 按分组缓存账号列表，供批量动作跨组扫描并按 ID 回查账号信息 |
| `emailListCache` | 邮件缓存（按 `email + folder`） |
| `currentAccount` | 当前右侧正在查看的账号 |
| `currentFolder` | 当前右侧打开的 folder |
| `currentEmails` | 当前右侧邮件列表数据 |

### 3.2 账号解析

批量动作开始时，前端需要从 `selectedAccountIds` 解析出目标账号清单。

实际解析方式应贴近当前代码结构：

- `accountsCache` 不是按账号 ID 直接建索引
- 而是按分组缓存账号数组
- 因此前端需要扫描 `Object.values(accountsCache)` 下的账号数组，再按 `id` 命中目标账号

最小必要字段：

1. `id`
2. `email`
3. `account_type`
4. `provider`

默认行为：

- 延续当前批量操作语义，允许处理跨分组的所有已选账号

### 3.3 请求数据流

对每个目标账号，前端依次或极小并发发起两次请求：

1. `GET /api/emails/<email_addr>?folder=inbox&skip=0&top=10`
2. `GET /api/emails/<email_addr>?folder=junkemail&skip=0&top=10`

设计意图：

1. latest-only
2. 更贴近验证码/最新邮件场景
3. 避免退化成整箱同步

### 3.4 成功回写

单个请求成功后，前端将结果写入：

`emailListCache[${email}_${folder}] = { emails, has_more, skip, method }`

如果返回包含 `account_summary`，则继续复用已有同步能力，将摘要回写账号缓存。

### 3.5 当前视图刷新条件

只有满足以下条件时才刷新当前右侧视图：

1. 本次处理账号 `email === currentAccount`
2. 本次处理 folder `=== currentFolder`

否则仅更新缓存，不打断当前浏览上下文。

## 4. 执行策略设计

### 4.1 V1 执行策略

V1 不新增后端 selected fetch 接口，直接在前端编排现有请求。

建议策略：

1. 串行为默认
2. 或保留极小并发（上限 1~2）

### 4.2 这样设计的原因

1. 当前项目存在单 worker / 单进程约束
2. 如果新增后端长时间阻塞式批量接口，容易出现“功能能跑，但整体不好用”
3. V1 目标是轻量可用，不是一次性做重型任务系统

### 4.3 失败处理

单个账号或单个 folder 失败时：

1. 记录失败项
2. 不中断整体批次
3. 最终在完成提示中汇总展示

## 5. 接口复用契约

### 5.1 复用接口

V1 复用现有接口：

`GET /api/emails/<email_addr>`

使用到的参数：

| 参数 | 值 |
|------|----|
| `folder` | `inbox` / `junkemail` |
| `skip` | `0` |
| `top` | `10` |

### 5.2 使用到的响应字段

| 字段 | 用途 |
|------|------|
| `success` | 判断当前请求是否成功 |
| `emails` | 写入 `emailListCache` |
| `method` | 标记缓存来源 |
| `has_more` | 保持与现有缓存结构一致 |
| `account_summary` | 如存在则同步回账号缓存 |

### 5.3 V1 不新增接口

本期明确：

- 不新增 `/api/accounts/fetch-selected`
- 不新增 SSE 流式 selected fetch 接口

## 6. 方案边界与后续演进

### 6.1 V1 边界

V1 的产品价值是：

1. 显式批量动作
2. latest-only 轻量拉取
3. 预热后续查看体验

而不是：

1. 邮箱批量同步系统
2. 多账号统一收件箱
3. 后端长任务编排中心

### 6.2 V2 演进方向

如果后续发现：

1. 选择账号很多时前端编排体验仍不够好
2. 需要可审计的任务态
3. 需要更平滑的后台推进

则下一阶段优先考虑：

1. 创建批量拉取任务
2. 返回任务 ID
3. 调度器/job/probe 推进
4. 前端轮询任务状态

即：优先走 **job / probe 化**，而不是直接扩展长时间 SSE。
