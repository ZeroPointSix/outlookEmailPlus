## 1. 文档目标

本 TDD 只关注 Issue #55 的测试设计，不重复展开产品背景与技术方案。

本次测试要证明六件事：

1. 标准模式批量操作栏已正确暴露“批量拉取邮件”入口
2. 前端批量拉取逻辑确实采用 **latest-only + `inbox/junkemail`** 的执行模型
3. 拉取成功后结果会正确回写现有缓存，而不是新建一套平行状态
4. 当前账号在批量完成后不会被自动切换
5. 单个账号或单个 folder 失败时，不会中断整批任务
6. V1 不新增后端批量接口的前提在测试上被保持，不会误把需求扩成新 API

---

## 3. 测试原则

1. **先测前端契约，再测行为**：按钮、入口函数、文案不存在，行为测试没有意义
2. **V1 不测不存在的新接口**：本期不新增 selected fetch API，因此不编写虚构接口测试
3. **缓存回写是核心断言**：这个功能的价值来自“预热现有状态”，而不是接口单次成功
4. **不以自动切换为成功标志**：当前账号不变是需求的一部分，必须显式断言
5. **失败不中断必须单独覆盖**：部分账号失败、部分 folder 失败都要继续推进整批任务

5. `currentAccount` 在完成后保持原值
