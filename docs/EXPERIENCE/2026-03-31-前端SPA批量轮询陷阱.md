# 经验积累：前端 SPA 批量轮询陷阱

**记录日期**：2026-03-31  
**关联 BUG**：`docs/BUG/2026-03-31-自动登录批量拉取邮件BUG.md`  
**关联提交**：`f83ded7`, `ee3b436`  
**技术标签**：JavaScript、SPA、轮询、调试、缓存

---

## 场景描述

在一个 Flask + 原生 JS 的单页面邮件管理应用中，用户登录后或点击导航进入邮箱页面时，系统自动对当前分组下**所有账号**批量发起了邮件拉取请求（N × 4 个并发请求），导致服务器 500 错误和邮件获取失败。

---

## 核心教训

### 1. SPA 中全局标志位非常不可靠

**场景**：想用 `_initialLoadComplete` 标志位区分"首次加载"和"后续操作"，首次不批量启动轮询。

**陷阱**：在 SPA 中，`DOMContentLoaded` 只触发一次，页面切换通过 `navigate()` 函数实现。`_initialLoadComplete` 在 Dashboard 加载后就被设为 `true`，导致后续切换到邮箱页面时仍然触发批量轮询。

**正确做法**：通过**函数参数**传递意图，而不是全局状态标志。或者直接从调用方决定是否执行某个行为。

```javascript
// ❌ 不可靠：全局标志在 SPA 中容易提前被污染
let _initialLoadComplete = false;
if (_initialLoadComplete && pollEnabled) { startBatchPoll(); }

// ✅ 可靠：通过参数明确传递意图
async function loadAccountsByGroup(groupId, forceRefresh = false, startPolling = false) {
    if (startPolling && pollEnabled) { startBatchPoll(); }
}
```

---

### 2. 添加参数时必须检查所有调用者——尤其是 HTML 内联 onclick

**场景**：给 `selectGroup(groupId)` 添加了第二个参数 `autoStartPolling = true`，打算在自动选中时传 `false`。

**陷阱**：`main.js` 中 Dashboard 的分组点击使用了**内联字符串生成的 onclick**：
```javascript
// main.js 中动态生成的 HTML
const clickAction = `navigate('mailbox'); setTimeout(function(){ selectGroup(${g.id}); }, 100)`;
```
这里 `selectGroup` 只传了 `groupId`，新增的 `autoStartPolling` 默认 `true`，修复完全失效。

**正确检查清单**：修改函数签名后，必须搜索所有调用者，特别关注：
- JS 代码中的直接调用 ✅（容易发现）
- `HTML` 模板中的 `onclick="funcName(arg)"` ✅（容易发现）
- `innerHTML` / `template literal` 中动态生成的 HTML 字符串 ⚠️（容易遗漏！）
- `setTimeout`/`setInterval` 回调中的调用 ⚠️（容易遗漏！）

**经验**：添加函数参数前，用全局搜索 `funcName\(` 确认所有调用场景。

---

### 3. "渐进式加条件守卫"不如"彻底删除不需要的逻辑"

**场景**：`loadAccountsByGroup` 中有批量启动轮询的逻辑，尝试通过添加 `_initialLoadComplete`、`startPolling`、`autoStartPolling` 等条件守卫来控制触发时机。

**陷阱**：总有遗漏的调用路径。每次修复一个漏洞，又发现另一个。

**根本原因**：**批量轮询在任何场景下都不应该触发**——这是一个本应从一开始就不存在的功能。

**正确做法**：如果一个行为在所有场景下都不应该发生，直接删除它，而不是通过条件守卫来"管控"它。

```javascript
// ❌ 3 次错误尝试：加条件守卫
if (_initialLoadComplete && ...) { batchStartPoll(); }
if (startPolling && ...) { batchStartPoll(); }
if (autoStartPolling && ...) { batchStartPoll(); }

// ✅ 正确方案：直接删除批量轮询逻辑
// 轮询只在 selectAccount（用户选中单个账号）时启动
```

**决策树**：
- 某行为在**任何场景**都不需要 → 直接删除
- 某行为在**部分场景**需要 → 参数控制，并确认所有调用者
- 某行为在**所有场景**都需要 → 无条件执行

---

### 4. 浏览器缓存是修复验证的隐形障碍

**场景**：修改了 JS 文件，刷新浏览器后发现问题依旧，以为修复没效果。

**陷阱**：浏览器缓存了旧版本 JS 文件，即使修改了服务器文件，浏览器仍然执行旧代码。

**解决方案**：
1. **Cache busting**：在 JS 文件 URL 后加版本参数 `?v=1.10.2`，每次修改后手动升级版本号
2. **硬刷新**：`Ctrl+Shift+R`（强制绕过缓存重载）
3. **开发者工具 Network**：勾选"Disable cache"（仅在 DevTools 打开时有效）

**本项目方案**：`templates/partials/scripts.html` 中所有 JS 文件加 `?v={{ APP_VERSION }}`，修改 `outlook_web/__init__.py` 中的 `__version__` 来触发全局缓存刷新。

```html
<script src="{{ url_for('static', filename='js/main.js') }}?v={{ APP_VERSION }}"></script>
```

```python
# outlook_web/__init__.py
__version__ = "1.10.2"  # 修改 JS 后同步升级这里
```

---

### 5. `console.trace()` 是定位"谁调用了这个函数"的最快方式

**场景**：`startPoll` 被调用了，但不知道调用链是哪条路径触发的。

**普通做法**：`console.log('startPoll called')` → 只知道被调用了，不知道从哪来

**正确做法**：`console.trace('startPoll called for:', email)` → 直接打印完整调用栈

```javascript
function startPoll(email, opts) {
    console.trace('[DEBUG] startPoll call stack for:', email);
    // ...
}
```

**输出示例**（浏览器控制台）：
```
[DEBUG] startPoll call stack for: user@example.com
  startPoll @ poll-engine.js:251
  (anonymous) @ groups.js:193
  loadAccountsByGroup @ groups.js:191
  selectGroup @ groups.js:159
  (anonymous) @ (index):1        ← 内联 onclick 触发！
  onclick @ (index):1
```

通过这个调用栈，立刻就能看出是 Dashboard 的内联 `onclick` 触发了 `selectGroup`，进而批量启动轮询。

## 修复时间线

| 时间 | 操作 | 结果 |
|------|------|------|
| 第1次尝试 | 添加 `_initialLoadComplete` 标志 | ❌ Dashboard 加载后标志提前变 true |
| 第2次尝试 | 改用 `startPolling` 参数（默认 false） | ❌ `selectGroup` 仍传 true |
| 第3次尝试 | 给 `selectGroup` 加 `autoStartPolling` 参数 | ❌ 内联 onclick 无法传参 |
| 添加 `console.trace` 日志 | 定位到内联 onclick 调用路径 | ✅ 找到根因 |
| 第4次（最终）修复 | 彻底删除 `loadAccountsByGroup` 中的批量轮询逻辑 | ✅ 问题解决 |

**总结**：前3次修复尝试都花了约20分钟，最终解决花了5分钟。根本原因是前3次在"如何限制这个行为"上思考，而正确方向是"这个行为根本不应该存在"。

## 可复用的调试模板

当遇到"某函数被意外调用"类问题时，可以在函数入口临时添加：

```javascript
function suspiciousFunction(arg) {
    console.warn('[DEBUG] suspiciousFunction called with:', arg);
    console.trace('[DEBUG] call stack');
    // ... 原有逻辑
}
```

定位到调用链后，针对性修复，然后移除调试日志。
